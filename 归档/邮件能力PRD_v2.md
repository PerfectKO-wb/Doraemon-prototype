## 一、背景

### 1.1 D 系统 3.0 的定位变化

D 系统 3.0 从"通讯与 AI 对话工具"转型为**"人-AI Agent-人"控制链的基础设施**。核心转变：

| 维度 | 2.0 定位 | 3.0 定位 |
|------|---------|---------|
| 系统角色 | 通讯 & AI 对话工具 | AI 任务管理基础设施 |
| AI 定位 | 被动响应用户指令 | 主动分派、监控、督促、验收 |
| 用户关系 | 用户使用工具 | AI Agent 管理用户完成工作 |
| 组织方式 | 按功能模块（邮件、聊天、文档…） | 按角色视图（执行者、决策者、维护者） |

### 1.2 邮件助手 v1 的问题

v1 将邮件助手设计为独立应用，存在以下与 3.0 架构的冲突：

| v1 假设 | 3.0 现实 | 冲突 |
|---------|---------|------|
| 独立应用入口 | 执行者的核心入口是 Worker 视图 | 多入口分散注意力 |
| 用户主动进入查看 | AI 应主动推送信息到用户面前 | 被动等用户来看 ≠ 主动触达 |
| 三 Tab 架构 | Worker 视图已有任务播报、统计、督促能力 | 功能重叠 |
| 通过跳转 Chat 完成操作 | Worker 视图内直接完成交互闭环 | 多次跳转破坏效率 |
| 面向所有用户无差别 | 三类角色有不同需求 | 缺少角色适配 |
| 仅面向邮件场景 | 邮件只是 Agent 的众多 Tool 之一 | 独立应用过重 |

### 1.3 邮件面向所有角色

**邮件不是某个岗位的专属功能，而是 D 系统的基础 Agent Tool。** 所有员工（执行者、决策者、维护者）都有邮件处理需求：

| 角色 | 邮件场景举例 |
|------|------------|
| CS 执行者 | 客户邮件回复、工单关联邮件通知 |
| BD 执行者 | 合作洽谈邮件、合同确认邮件 |
| 运营执行者 | 供应商邮件、活动协调邮件 |
| 决策者 | 审批流邮件、汇报邮件、外部高层邮件 |
| 维护者 | 系统告警邮件、平台通知邮件 |

## 二、设计目标

| 目标 | 说明 |
|------|------|
| 邮件能力融入 3.0 架构 | 不再作为独立应用，拆散到三个角色视图中，以最合适的形态呈现 |
| 邮件 = Agent 的底层 Tool | AI Agent 通过邮件 Tool 感知和操作邮箱，各角色视图按需调用 |
| 定时批量拉取 | 每日两个批次（08:30 / 13:00），邮件任务数量有限、可清零、可结算 |
| 维护者通过 Chat 配置 | 工作流和 Agent 行为通过自然语言对话创建和修改 |
| 执行者无自助配置 | 一线执行者专注处理任务，不自行配置自动化规则 |

## 三、核心设计决策

### 3.1 采用方案 C：邮件能力拆散到角色视图

不再存在"邮件助手"这个独立概念。邮件能力被拆解为 AI Agent 的底层 Tool，根据三类角色在各自视图中呈现：

```
┌─────────────────────────────────────────────┐
│           邮件 Tool（底层能力）                │
│  收发邮件 / 分析内容 / 打标签 / 归档 / 搜索    │
└──────────┬──────────────┬──────────────┬─────┘
           │              │              │
    ┌──────▼──────┐ ┌─────▼──────┐ ┌────▼──────┐
    │  执行者视图   │ │  决策者视图  │ │  维护者视图 │
    │   Worker    │ │  Dashboard │ │   Works   │
    │  邮件任务卡片 │ │  邮件效率指标 │ │  工作流配置 │
    │ + inline 操作│ │ + 积压告警  │ │ + Agent管理│
    └─────────────┘ └────────────┘ └───────────┘
```

### 3.2 定时批量拉取（非实时）

**核心问题**：如果 Agent 每 2 分钟拉取邮件，执行者一天都有处理不完的邮件，无法触发"任务全部完成"的结算场景。

**解决方案**：每日固定两个批次拉取

```
批次 1：每日 08:30（上班前）
  → Agent 拉取昨日下班后至今晨的所有新邮件
  → AI 批量分析 → 需要人工处理的进入「邮件待处理队列」
  → 纳入早晨播报：📧 邮件处理 · N 封待处理

批次 2：每日 13:00（下午上班前）
  → Agent 拉取上午新到达的邮件
  → AI 分析 → 追加到队列
  → 推送补充通知：「新增 M 封邮件需要处理」
```

| 维度 | 效果 |
|------|------|
| 任务有限 | 每个批次的邮件数确定，可以清零 |
| 可结算 | Output 可以写「上午处理 X 封 / 下午处理 Y 封」 |
| 节奏感 | 和员工工作节奏匹配（上午一批、下午一批） |
| 播报自然 | 早晨播报自然包含邮件任务数 |
| Token 省 | 批量分析比逐封分析效率高 |

**紧急邮件通道（V2 规划）**：
- MVP 只做定时批量，不做实时
- V2 增加轻量级实时监控通道：仅对 VIP 发件人 + 紧急关键词做实时推送
- 实时通道不做 AI 全量分析，仅做关键词/发件人匹配

### 3.3 维护者通过 Chat 创建工作流和 Agent 配置

延续 v1 邮件助手 Tab 3 的 Chat 规则创建方案，升级后应用到维护者视图：

| 维度 | v1 邮件助手 Tab 3 | 3.0 维护者 Works |
|------|-----------------|-----------------|
| 配置者 | 一线员工自己 | 维护者（专业角色） |
| 配置范围 | 仅邮件规则 | 工作流 + Agent 行为 + 跨系统联动 |
| 配置对象 | 只影响自己 | 影响团队/全员 |
| 应用范围 | 无 | 基于 P 系统所属团队，维护者只能配置其管辖范围内的团队 |
| 修改方式 | 重新对话 | 点击已有工作流 → Chat 面板加载历史对话，继续修改 |

### 3.4 执行者无自助配置能力

一线执行者不具备自行配置自动化规则的能力。所有邮件相关的自动化规则由维护者统一管理。

## 四、邮件监控 Agent

### 4.1 触发机制

| 配置项 | 默认值 | 说明 |
|--------|-------|------|
| 拉取模式 | 定时批量 | 非实时 |
| 拉取时间 | 08:30 / 13:00 | 员工本地时区，维护者可调整 |
| 监控范围 | 基于 P 系统团队配置 | 维护者按团队设定 |
| 分析模式 | 批量 AI 分析 | 同一批次内的邮件可做关联判断 |

### 4.2 分析策略

邮件监控 Agent 对每个批次的邮件执行以下分析流程：

```
新邮件批次输入
  ↓
1. 发件人识别（匹配 P 系统客户/联系人库）
  ↓
2. 内容分类
   · 纯通知/营销类 → 自动归档，不推送
   · 系统通知（OA/监控）→ 自动归档 + 打标签
   · 需要回复的商务邮件 → 生成回复草稿，进入人工队列
   · 包含审批/决策内容 → 推送到决策者
   · 无法判断 → 推送到邮件所有者
  ↓
3. 优先级判定
   · 紧急：截止日期 ≤ 3 天 / VIP 发件人 / 邮件标记重要
   · 一般：其他商务邮件
  ↓
4. 执行者匹配（按 P 系统客户负责人关系）
  ↓
5. 推送到对应角色的视图
```

### 4.3 任务分发逻辑

| 分发目标 | 匹配规则 | 推送方式 |
|---------|---------|---------|
| 执行者 | P 系统客户负责人关系 | 纳入 Worker 视图任务卡片 |
| 决策者 | 邮件内容含审批/决策关键词 | 纳入 Dashboard 待办 |
| 邮件所有者 | 无法匹配的兜底 | 纳入其 Worker 视图 |

## 五、执行者视图（Worker）

### 5.1 邮件作为任务类型

邮件处理作为 Worker 视图中的一种任务类型，与其他岗位任务并列展示：

```
<task-cards>
<task id="email" pending="4" />        ← 邮件任务（所有岗位通用）
<task id="contracts" pending="3" />     ← BD 岗位示例
<task id="proposals" pending="2" />     ← BD 岗位示例
</task-cards>
```

不同岗位的 Worker 视图中，邮件任务卡片均会出现，其他任务卡片根据岗位 Skill 不同而不同。

### 5.2 早晨播报中的邮件任务

早晨播报包含批次拉取信息和邮件任务摘要：

```
⏰ 08:30 早晨批次 · 拉取昨晚至今晨新邮件 18 封 → AI 分析完成，4 封需要你处理

早上好 ☀️
今天有 3 项任务需要处理，其中有 4 封邮件需要你回复。

<task-cards>
<task id="email" pending="4" />
<task id="contracts" pending="3" />
<task id="proposals" pending="2" />
</task-cards>

<task-ref id="email" />
有 1 封来自田中太郎（A 出版社）的邮件标记为紧急，建议优先处理。
```

### 5.3 邮件 Inline Action

邮件任务在 Worker 视图中以 inline action 形式展示，每封邮件包含：

| 区域 | 内容 |
|------|------|
| 邮件头 | 发件人 + 优先级标签 + 时间 |
| 主题 | 邮件主题 |
| 摘要 | AI 生成的一句话摘要 |
| AI 草稿 | AI 生成的回复草稿（展示在紫色边框区域内） |
| 操作按钮 | 编辑回复 / 直接发送 / 稍后处理 |

跨系统操作以独立的 action 卡片展示在邮件下方：

```
[A] 创建任务「天剑监修反馈处理」→ 指派 David Chen, DDL 3/28    [确认] [跳过]
[📅] 创建提醒：周五前回复田中 → 2026/03/27 09:00              [确认] [跳过]
```

### 5.4 邮件详情侧边面板

点击"编辑回复"或邮件任务卡片时，弹出邮件详情面板，包含：

| 区域 | 内容 |
|------|------|
| 邮件元信息 | 发件人、主题、时间、优先级 |
| 原始邮件 | 完整邮件原文 |
| AI 草稿 | 可编辑的回复草稿 |
| 关联操作 | AI 建议的跨系统操作列表 |
| 底部操作栏 | 编辑并发送 / 直接发送草稿 / 稍后处理 |

### 5.5 下午批次推送

13:00 批次拉取完成后，Worker 视图中追加一条补充推送：

```
⏰ 13:00 下午批次 · 拉取今日上午新邮件 6 封 → AI 分析完成，2 封需要你处理

下午好 👋
13:00 批次拉取完成，上午新增 2 封邮件需要你处理。
```

| AI 自动 | AI 自动处理数量 |

## 七、维护者视图（Works）

### 7.1 整体布局

维护者视图采用**两栏布局**：左侧为列表面板，右侧为 Chat 面板。

顶部设三个 Tab：工作流 | Agent 配置 | 执行日志

### 7.2 工作流管理

#### 7.2.1 工作流列表（左侧面板）

每条工作流包含：

| 元素 | 说明 |
|------|------|
| 状态指示灯 | 绿色 = 运行中，灰色 = 已暂停 |
| 工作流名称 | AI 自动命名或维护者自定义 |
| 类型标签 | 邮件 / 跨系统 |
| 触发信息 | 触发频率 + 应用范围 |

点击工作流项，右侧 Chat 面板加载该工作流的创建/修改对话历史。

顶部设"+ 新建"按钮，点击后右侧 Chat 面板清空，AI 引导维护者描述新工作流。

#### 7.2.2 Chat 创建工作流（右侧面板）

维护者通过自然语言描述工作流需求，AI 解析后生成工作流摘要卡片。

**Chat 交互流程：**

```
维护者: 设置一个邮件处理流程，每天早上8:30和下午1点拉取新邮件，
        AI分析后把需要人工处理的推给对应执行者
   ↓
AI: 有几个问题需要确认：
    1. 邮件分发规则：按什么逻辑匹配执行者？
    2. 自动归档条件：哪些邮件不需要推送？
    3. 应用范围：覆盖哪些团队？
   ↓
维护者: 按P系统客户负责人匹配，通知类自动归档，应用到全公司
   ↓
AI: [展示 P 系统可选团队范围]
   ↓
AI: [生成工作流摘要卡片]
   ↓
维护者: 点击「确认创建」
   ↓
AI: 工作流已成功创建并启动！
```

#### 7.2.3 工作流摘要卡片

AI 在对话中生成的配置摘要，包含以下字段：

| 字段 | 说明 |
|------|------|
| 名称 | AI 自动命名 |
| 触发条件 | 定时 / 实时 / 事件触发 |
| 应用范围 | 全公司 / 特定团队（基于 P 系统） |
| 执行策略 | 触发条件 + 执行动作的完整描述 |
| 确认模式 | 自动执行 / 待确认后执行 |
| 操作按钮 | 确认创建 / 继续编辑 |

#### 7.2.4 应用范围（基于 P 系统）

维护者创建工作流时，可选的应用范围由其在 P 系统中的所属团队决定：

- 系统读取维护者在 P 系统的管辖团队列表
- 在 Chat 中以可选标签形式展示
- 维护者选择后，工作流仅对该范围内的员工生效

#### 7.2.5 修改已有工作流

点击左侧列表中的工作流项，右侧 Chat 面板加载创建时的对话历史。维护者可以在对话中继续追加修改指令：

```
维护者: 把触发时间改成 9 点
AI: 好的，触发时间已从 08:30 更新为 09:00。
    [更新后的摘要卡片]
```

#### 7.2.6 字段采集规则

延续 v1 的字段采集逻辑，AI 在对话中需要收集以下字段：

| 字段 | 必填性 | 说明 |
|------|--------|------|
| 触发条件 | 必填 | 什么情况下触发 |
| 执行动作 | 必填 | 触发后要做什么 |
| 触发频率 | 必填 | 多久检查一次 / 什么时候执行 |
| 确认模式 | 必填 | 执行前是否需要确认 |
| 应用范围 | 必填 | 覆盖哪些团队（基于 P 系统） |
| 目标系统 | 条件必填 | 涉及跨系统操作时必须明确 |
| AI 执行策略 | 条件必填 | 对 AI 执行有特殊要求时需说明 |

缺少信息时，AI 主动追问（追问话术延续 v1 PRD 2.3.3 节的设计）。

### 7.3 Agent 配置管理

#### 7.3.1 Agent 列表（左侧面板）

展示所有邮件相关的 Agent：

| Agent | 说明 |
|-------|------|
| 邮件智能分类 & 行动生成 | 分析收件内容，判断优先级，生成回复草稿和跨系统操作建议 |
| 邮件监控 Agent | 定时拉取邮箱新邮件并触发工作流 |

#### 7.3.2 Chat 编辑 Agent 行为

点击 Agent 项后，右侧 Chat 面板展示当前配置概要，维护者可通过自然语言修改：

```
AI: 当前「邮件智能分类」Agent 配置如下：
    · 分析策略：批次分析所有新收件
    · 回复语言：自动识别原文语言
    ...
    你想修改哪些配置？

维护者: 日本客户的回复用正式商务日语

AI: 好的，确认更新：
    · 日本客户邮件 → 正式商务日语
    · 其他 → 中文
    [Agent 配置摘要卡片]
```

#### 7.3.3 可配置参数

| 参数 | 说明 | 默认值 |
|------|------|--------|
| 分析策略 | 批量分析 / 逐封分析 | 批量分析 |
| 优先级判定规则 | 紧急条件定义 | 截止日期 ≤ 3 天 / VIP / 标记重要 |
| 回复语言策略 | 按发件人/域名匹配语言 | 自动识别原文语言 |
| 跨系统联动 | 启用/禁用各系统联动 | 全部启用 |
| 自动处理阈值 | 连续确认 N 次后提议自动执行 | 10 次 |
| 邮件模板 | 回复署名模板 | 公司统一模板 |

### 7.4 执行日志

单栏展示（无 Chat 面板），以时间倒序列出邮件 Agent 的执行记录：

| 字段 | 说明 |
|------|------|
| 时间 | 执行时间点 |
| 描述 | 执行的操作描述（批次拉取、邮件分析、自动归档、播报推送等） |
| 状态 | 完成 / 失败 / 进行中 |

## 八、A 票集成与 Output 结算

### 8.1 邮件处理作为子任务

邮件处理纳入每日 A 票体系，作为一种通用子任务类型（不依赖岗位 Skill）：

```
A 票标题格式：
· 邮件处理（所有岗位通用）
· 客服工单处理（CS 岗位）
· 合同跟进（BD 岗位）
· ...
```

每日自动创建 A 票时，如果该员工当天有邮件待处理（经过 08:30 批次分析后），则自动创建「邮件处理」A 票。

### 8.2 Output 结算格式

每日 00:00 结算时，邮件处理的 Output 格式：

```
■ 邮件处理
  处理邮件总数：{N} 封
  · 确认 AI 草稿直接发送：{a} 封
  · 编辑后发送：{b} 封
  · 标记不处理：{c} 封
  跨系统联动：创建 A 任务 {x} 个 · 日历提醒 {y} 个
  批次数据：早晨批次 {m} 封 · 下午批次 {n} 封
```

## 九、跨系统集成

延续 v1 的跨系统能力，在 3.0 架构中作为 Agent Tool 调用：

| 系统标识 | 系统名称 | 联动能力 |
|---------|---------|---------|
| @邮件 | 邮件系统 | 发送/回复/转发邮件、打标签、归档 |
| @A | A-System（任务管理） | 创建任务、查询任务状态、检测逾期 |
| @P | P-System（人员架构） | 查询员工信息、客户负责人关系、团队归属 |
| @D | D-System（通讯系统） | 发送消息、转发联系人信息 |
| @日历 | 日历系统 | 创建提醒、创建日程 |

**联动方向：**

| 方向 | 说明 | 示例 |
|------|------|------|
| 邮件 → 外部系统 | AI 分析邮件后主动操作外部系统 | 收到合作邮件 → 创建 A 任务 |
| 外部系统 → 邮件 | 外部系统事件触发邮件操作 | A 任务逾期 → 生成催促邮件草稿 |
