# D 系统 3.0 — A 组工作总结

---

## 一、D 系统 3.0 的新定位

D 系统从**"被动响应的通讯与 AI 对话工具（2.0）"**，彻底转型为**"人-AI Agent-人 控制链的基础设施（3.0）"**。

最本质的变化是：**从"人管任务"变成"AI 管任务"**。

| | 老模式（2.0） | 新模式（3.0） |
|---|---|---|
| **任务创建** | 员工每天自己在 A 系统创建 A 票 | AI 自动创建，从各业务系统拉数据生成任务清单 |
| **进度跟踪** | 员工自己更新状态，主管手动检查 | AI 实时监控，做完自动划掉，没动静就催 |
| **日报/结算** | 员工手写日报，主管逐条核对 | AI 自动结算，汇总完成情况生成 A 票记录 |
| **规则配置** | 提需求给开发，排期上线 | 维护者用自然语言聊天配置，即时生效 |

**一句话：以前员工每天第一件事是"开 A 票、写计划"，以后第一件事是"打开 D 系统，看 AI 给我安排了什么"。**

---

## 二、A 组做什么：AI 引擎基建

A 组不直接做业务功能，而是做**所有业务 Agent 共享的底层 AI 能力**。可以理解为：B 组造车，A 组修路。

A 组的工作方向分为三大块：

| 方向 | 说明 | 具体内容 |
|------|------|---------|
| **引擎基建** | AI 运行的核心底层能力 | 意图路由、Skill 标准体系、记忆系统、输出格式校验 |
| **性能优化** | 让 AI 快且准 | TTSR 响应速度优化、按需加载 Prompt/MCP、可插拔意图管理 |
| **内部自动化** | 提升团队自身效率 | E2E 自动化测试、入职流程自动化（一键入职） |

### Phase 1：CS 打样（Sprint 43，3/23 - 4/3）—— B 组主导，A 组不介入

> **先用 CS 一个岗位跑通"AI 给员工派任务、盯进度、收结果"的完整流程。**

这个阶段由 B 组用硬编码引擎跑通 CS 场景，验证"AI 管任务"这个模式是否成立。**A 组不参与此阶段。**

Phase 1 完成后交付物：
- CS 岗位的 Worker 视图（晨播报、督促、完成、下班告警）
- 硬编码的 CS 任务 Agent（从 A 系统拉数据、自动结算）
- 最小引擎（定时触发 + Output Parser）

---

### Phase 2：引擎通用化 + 接入新岗位（Sprint 44-47，4/6 - 5/29）

> **把 Phase 1 的硬编码引擎改造成通用架构，然后用这套架构接入翻译官和画师。**

**总体方向：Phase 1 的引擎只能服务 CS 一个岗位，Phase 2 要让引擎变成"能接任何岗位"的通用架构——B 组可以用标准 SDK 快速开发新岗位的 Skill，不用每次从零开始。前半段（Sprint 44-45）打地基，后半段（Sprint 46-47）用新引擎接入翻译官和画师。**

翻译官和画师都是"生产型岗位"——任务输入清晰、交付物可衡量、流程可标准化。先啃这类岗位验证引擎通用性，再过渡到更复杂的岗位。

**A 组需要完善的能力及原因：**

| 能力 | 为什么 |
|------|--------|
| Skill 标准 & SDK | 没有标准，B 组接每个新岗位都要从零开发。这是所有后续工作的地基——标准定不好，后面全返工 |
| AIU 迁移为标准 Skill | 现有 AIU 能力要标准化，否则接新岗位后 Phase 1 的功能会退化 |
| 记忆系统 V1 | 不同岗位需要记住不同的上下文信息——没有记忆每次都要重复说明。先在 CS 上验证可行性 |
| TTSR 基线测量 + 优化 | 不测基线就优化等于盲人摸象。多岗位同时在线后并发翻倍，必须先知道现在有多慢 |
| Tool Gateway 扩展 | 新岗位的任务来自不同的外部系统，必须先把数据管道打通 |
| 新岗位 Skill 开发 | 用标准 SDK 给翻译官、画师开发专属 Skill |
| Chat-to-Config 引擎 | 维护者要用自然语言配工作流，需要后端引擎支撑 |
| 入职流程自动化 | 新岗位上线意味着新员工批量接入系统，手动配权限太慢 |

### Phase 3：扩展更多岗位 + 平台化（Sprint 48-55，6/1 - 9/18）

> **从 3 个岗位扩展到 UI、PM、QA 等更多岗位，Skill 从手写进化到自动生成，AI 开始有自主执行能力。**

**总体方向：前两个阶段的岗位都是"生产型"（任务输入清晰、可衡量），Phase 3 要接入"分析创造型"岗位（任务更开放、判断更主观），引擎必须更智能。同时 Skill 不能再全靠人手写，要能从数据中自动生成；AI 也要从"问一句做一步"进化到"某些操作连续确认 N 次后主动提议以后我直接做"。**

**A 组需要完善的能力及原因：**

| 能力 | 为什么 |
|------|--------|
| Skill 自动生成 | 岗位越来越多，每个都手写 Skill 来不及了，要从用户交互日志中发现可复用模式自动生成 |
| Skill 开放平台 | 不同岗位的 Skill 只有对应团队最懂，要让业务方也能创建和贡献 |
| 记忆系统 V2（长期记忆） | 部分岗位需要记住项目全生命周期上下文——V1 的工作记忆不够用 |
| Tool Gateway 继续扩展 | 新岗位需要接入更多外部系统 |
| AI 自主执行能力 | 岗位多了重复任务也多了，某些操作人连续确认 N 次后 AI 应该主动提议"以后这类我直接做" |
| TTSR 继续优化（目标 P90 < 1s） | 多岗位同时在线，Skill 和 Tool 数量翻倍，必须继续提速 |
| E2E 测试覆盖 80%+ | 岗位多了 Skill 多了，回归风险指数级增长，必须有自动化测试兜底 |
| 实时事件推送 | 所有岗位都有紧急场景，需要毫秒级推送不等批次 |
| Dashboard 数据聚合 | 决策者看板要跨岗位汇总，A 组提供标准数据接口 |
