# CS Agent Skill 产品设计思路

## 1. 背景与核心理念

在传统的客服 AI 方案中，我们通常依赖**知识库（Knowledge Base）**来解决问题。知识库的本质是"回答什么"（What to say），它提供事实、政策、FAQ 等静态信息。

然而，优秀的客服不仅仅是背诵规则，更需要**处理场景**。面对一个愤怒的 VIP 充值未到账用户，客服不仅需要知道"充值到账时间"，还需要知道：

1. 应该用什么语气安抚用户？
2. 应该先查单还是先让用户提供截图？
3. 如果查不到单怎么处理？如果查到单但没发货怎么处理？
4. 什么情况下需要直接转接给主管？

这就是 **Skill（话术/场景能力）** 解决的问题。Skill 的本质是"怎么回答 + 下一步做什么"（How to say & What to do）。

**核心理念：**

- **Knowledge 是知识的存储层**，负责提供事实支撑。
- **Skill 是能力的执行层**，负责封装业务策略、操作工具和行为约束。
- 面对玩家提问，Agent 首先匹配对应的 Skill 场景，在 Skill 的指导下（结合精准引用的知识）完成对话和操作。

---

## 2. 一个 Skill 包含哪些内容？

为了让 Agent 能像优秀客服一样处理特定场景，一个完整的 Skill 包含以下 6 个维度的内容：

### ① 触发条件 (Trigger) —— 何时使用这个 Skill？

定义在什么情况下 Agent 应该激活这个能力。

- **用户意图**：如"充值未到账"、"账号被封"。
- **关键词**：如"没收到"、"扣款了"、"解封"。
- **适用条件**：针对特定用户群体（如仅限 VIP）或特定游戏（如仅限 vividarmy）。

### ② 处理指令 (Instructions) —— 怎么处理这个问题？

给 Agent 的直接行为指导，相当于"话术模板"的智能升级版。

- **处理目标**：如"快速定位未到账原因，优先保障体验"。
- **语气要求**：如"共情、专业、高优先级"。
- **SOP 步骤**：明确第一步做什么（安抚），第二步做什么（查单），第三步做什么（根据状态分流）。
- **处理原则**：如"先给方案再解释原因"、"不要让用户等"。

### ③ 适用工具 (Tools) —— 能执行什么操作？

Agent 在处理该场景时可以调用的系统能力。

- 如：`查询订单状态`、`创建补发工单`、`转接人工客服`、`给会话打标签`。
- Agent 会根据对话进展，自动决定何时调用哪个工具，并在界面上展示执行结果。

### ④ 引用知识 (Knowledge Refs) —— 需要参考哪些事实？

Skill 执行时需要依赖的外部知识。

- 与其让 Agent 每次都在海量知识库中盲目搜索，不如在 Skill 中精准绑定需要的知识条目（如"充值到账时间说明"、"VIP 补偿政策"）。
- 这样既保证了知识的准确性，又提高了 Agent 的响应效率。

### ⑤ 约束与边界 (Guardrails) —— 绝对不能做什么？

防止 Agent 犯错的"红线"。

- 如："未确认订单状态前，绝不承诺补发或退款"。
- 如："金额超过 1000 的订单，必须转接专属团队，不可自动处理"。
- 如："绝对不要向用户透露内部系统名称"。

### ⑥ 参考案例 (Examples) —— 优秀客服是怎么做的？

提供真实的优秀历史对话作为示范。

- 包含玩家的提问、当时的上下文、客服的完美回复以及执行的操作。
- 帮助 Agent 更好地模仿人类专家的处理风格。

## 3. Skill 的生命周期与管理

Skill 不是一成不变的，它需要像人类客服一样不断学习和进化。

### 创建与学习

- **AI 自动学习**：系统自动分析历史优秀工单，提取出处理模式，生成 Draft（草稿）版本的 Skill。
- **Chat 方式创建**：管理员可以通过自然语言对话的方式，直接命令 AI 创建或修改 Skill（例如："帮我加一条约束：大额订单必须转人工"）。

### 审核与测试（沙盒）

在 Skill 正式上线前，必须经过严格的验证：

- **人工审核**：管理员检查指令是否准确、工具权限是否合理。
- **沙盒测试**：在管理后台输入模拟玩家消息，系统会直观地展示：
  - **匹配测试**：Intent 匹配度打分、关键词命中情况。
  - **回复测试**：Agent 的模拟回复及语气评分、步骤遵循情况。
  - **工具测试**：调用的工具及参数是否正确。
  - **约束检查**：是否严格遵守了所有 Guardrails（逐条打钩验证）。

### 版本控制与回退

- 每次对 Skill 的修改都会生成新的版本记录。
- 如果新上线的版本表现不佳，管理员可以一键回退到任意历史版本。
- 回退操作不会直接覆盖线上版本，而是基于历史版本生成一个新的草稿（Draft），重新经过审核和测试后才能上线，确保线上服务的绝对安全。
