## 一、背景

### 1. 现状

D-System 已通过 API 连接用户邮箱，并具备以下 AI 邮件能力：

- **即时处理**（问一次答一次）：生成回复（多语言）、提取关键信息、分类排序、自然语言搜索、跨文档综合分析
- **定时自动化**（设一次跑多次）：定时总结特定邮件、关键词触发提醒、按频率汇总报告
- **跨系统连接**：通过 MCP 已打通 A-System（任务管理）、P-System（人员架构）等业务系统

**关键约束：D-System 不是邮箱客户端，没有邮件阅读界面。** 用户日常仍在邮箱客户端（Outlook / Gmail）中处理邮件，D-System 的邮件能力只能通过聊天对话触发，用户感知不到 D-System 和邮箱之间的关系。

### 2. 痛点

| 痛点 | 说明 |
|------|------|
| 有能力但用户无感知 | D-System 已经能操控邮箱，但用户打开看到的是空聊天框，不知道 AI 能帮自己处理邮件 |
| 能力是被动的 | 所有操作（包括定时任务和关键词监控）都需要用户主动通过聊天触发，AI 在等着被吩咐，不是主动在工作 |
| 通知只是信息，不是行动 | 定时总结输出的是文字报告，用户看完还要自己去邮箱客户端处理 |
| 邮件和业务系统割裂 | 很多邮件的终点不是"回一封邮件"，而是去 A-System 建任务、查人员信息等。D-System 虽已通过 MCP 连接这些系统，但只在用户主动聊天时才触发，邮件事件不会自动联动其他系统 |

### 3. 解决思路

在 D-System 中构建**邮件助手**，将已有的邮件 AI 能力从「被动等吩咐」变为「主动帮用户干活」，让用户打开就能看到结果，而不是一个空聊天框。

核心链路：**AI 后台持续分析邮件 → 生成行动方案 → 用户在邮件助手中逐项确认/修改 → 通过 Chat 执行**

1. AI 后台持续拉取新邮件，自动分析并生成结构化行动方案（回复草稿、跨系统操作），用户打开即看到待处理事项
2. 用户对每个行动项独立决策（确认/修改/不处理），通过 Chat 页面完成交互闭环
3. 支持通过自然语言对话配置自动化规则，重复性操作由 AI 自动或半自动执行
4. 邮件事件可自动联动已接入的业务系统，业务系统事件也可触发邮件操作

## 二、目标

| 目标 | 说明 |
|------|------|
| 让用户感知到 D-System 的邮件能力 | 用户打开邮件助手就能看到 AI 已经分析好的邮件和待处理事项，而不是一个空聊天框 |
| 从被动变主动 | AI 后台持续工作，主动生成行动方案（回复草稿、跨系统操作建议），用户只需审核确认 |
| 邮件与业务系统联动 | 邮件事件可自动触发其他系统操作，其他系统事件也可触发邮件操作，用户不再需要手动切换系统 |
| 重复操作自动化 | 通过自然语言配置自动化规则，让 AI 自动处理归档、标签、感谢回复等重复性操作 |

## 三、需求

### 1. 功能架构

邮件助手采用三 Tab 架构：

| Tab | 名称 | 功能定位 |
|-----|------|---------|
| Tab 1 | 需要处理 | 展示 AI 分析后需要用户决策的邮件及规则触发任务 |
| Tab 2 | 已处理 | 展示用户已处理和 AI 自动处理的邮件记录及统计 |
| Tab 3 | 自动化规则 | 管理自动化规则列表 + 通过 Chat 创建新规则 |

页面入口：从 workspace.html 的「邮件助手」应用卡片进入，卡片上显示待处理数量红点。页面 Header 左侧有返回按钮可回到工作区。

Header 右侧显示今日邮件统计：今日收到总数、AI 已处理数、需你关注数。

### 2. 功能需求清单

#### 2.1 Tab 1：需要处理

##### 2.1.1 任务卡片

AI 分析所有收到的邮件后，将需要用户决策的内容以统一的**任务卡片**形式展示。所有类型的任务卡片采用相同的展示结构：

**卡片信息结构：**

| 层级 | 内容 | 说明 |
|------|------|------|
| 第一行（Meta） | 发件人/来源 + 来源标签 + 时间 + 卡片级操作 | 发件人为邮件发件人或触发来源描述 |
| 第二行（主题） | 邮件主题或任务标题 | 单行截断 |
| 第三行（摘要） | AI 生成的一句话摘要 | 单行截断 |
| 行动方案区域 | AI 生成的操作项列表 | 每项独立附带操作按钮 |
| 原文展开区（可选） | 原始邮件内容 | 点击「原文」展开/收起 |

**来源标签：**

所有任务卡片均由规则或 AI 分析触发，标签统一标注**数据来源**：

| 标签示例 | 含义 |
|----------|------|
| 来源：邮箱 | 通过分析邮件内容触发（包括 AI 智能识别和用户自定义邮件规则） |
| 来源：A | 由 A-System 事件触发 |
| 来源：D | 由 D-System 事件触发 |
| 来源：P | 由 P-System 事件触发 |

视觉统一：灰色背景（`--bg-active`）+ 次级文字色（`--text-secondary`），不做颜色区分。

**卡片优先级指示：** 通过卡片左侧边框颜色区分

| 类型 | 颜色 | 场景 |
|------|------|------|
| urgent | 红色（error-base） | 高优先级，如有截止日期、逾期等 |
| normal | 蓝色（cs-primary-color） | 普通优先级 |
| rule | 橙色（orange-base） | 规则触发的待确认任务 |

##### 2.1.2 AI 行动方案（Action Items）

每张卡片的核心区域。AI 为每封邮件生成一个或多个行动项，每个行动项代表一个独立可执行的操作。

**行动项结构：**

```
· [操作描述文本]                    确认  修改  不处理
```

每个行动项独立包含三个操作按钮：

| 按钮 | 样式 | 点击行为 |
|------|------|---------|
| 确认 | 蓝色文字链接（cs-primary-color），hover 变蓝底白字 | 跳转 chat.html，内容作为用户第一条消息直接发送（mode=confirm） |
| 修改 | 灰色文字链接（text-secondary），hover 变灰底深色字 | 跳转 chat.html，内容回填到输入框等用户编辑后发送（mode=edit） |
| 不处理 | 浅灰文字链接（text-disabled），hover 变红色（error-base） | 移除该条行动项，动画收起；若卡片内所有行动项均被移除，整张卡片自动消失 |

**跳转 Chat 时的内容格式：**

所有行动项跳转 Chat 时，发送/回填的内容统一采用以下格式：

```
@{系统标识} {动作指令}
{结构化参数}
```

示例：

| 行动类型 | 发送到 Chat 的内容 |
|---------|------------------|
| 回复邮件 | `@邮件 帮回复田中太郎`<br>`内容：田中先生，一直承蒙关照...` |
| 创建任务 | `@A 帮创建任务`<br>`内容：天剑监修反馈  指派：David Chen  DDL：2026/03/21` |
| 创建提醒 | `@日历 帮创建提醒`<br>`内容：周五前回复田中  时间：2026/03/21 09:00` |
| 转发联系人 | `@D 帮转发联系人信息`<br>`联系人：张伟 · ABC 公司 BD 总监  接收人：王明（BD 部门）` |
| 批量打标签 | `@邮件 帮批量打标签`<br>`命中邮件：3 封 · 均来自 A 出版社  标签：监修、天剑项目` |

##### 2.1.3 卡片级操作

位于卡片第一行（Meta 行）右侧，与时间字段同行：

| 按钮 | 说明 |
|------|------|
| 原文 | 展开/收起原始邮件内容（仅有原始邮件的卡片显示） |
| 全部不处理 | 移除整张卡片，所有行动项均不执行 |

##### 2.1.4 AI 授权建议（Trust Banner）【第二期】

> **本期不做，列入第二期规划。** 本期所有任务均需用户逐项确认。

当 AI 检测到用户对某类邮件存在重复处理模式时（如连续 N 次直接发送某类草稿），在卡片列表顶部展示授权建议横幅：

| 元素 | 说明 |
|------|------|
| 标题 | AI 授权建议 |
| 描述 | 说明识别到的行为模式，如「你过去 10 次都直接发送了感谢邮件草稿」 |
| 同意授权 | 后续该类邮件由 AI 自动处理，不再进入「需要处理」Tab |
| 暂不 | 关闭横幅，下次满足条件再提示 |

**触发策略：** 基于 Pattern Tracker 统计用户对同类邮件的重复确认行为，达到阈值后触发，而非每封邮件都询问。

##### 2.1.5 任务卡片的典型场景

| 场景 | 来源 | AI 生成的行动项示例 |
|------|------|-------------------|
| 外部合作方邮件需回复 | 来源：邮箱 | ① 回复邮件 ② 跨系统创建任务 ③ 日历创建提醒 |
| 外部系统触发的发件任务 | 来源：A | ① 发送催促/通知邮件 |
| 包含报价/合同的邮件 | 来源：邮箱 | ① 回复邮件 ② A-System 创建任务 |
| 命中关键词规则 | 来源：邮箱 | ① 跨系统转发联系人信息 |
| 命中自动标签规则 | 来源：邮箱 | ① 批量打标签 |


#### 2.2 Tab 2：已处理

展示所有已处理邮件的记录和统计。

##### 2.2.1 统计面板

页面顶部四格统计卡片：

| 统计项 | 说明 |
|--------|------|
| 今日收件 | 今日收到的邮件总数 |
| AI 自动处理 | AI 根据授权/规则自动处理的数量（绿色） |
| 你已处理 | 用户手动确认/编辑/发送的数量（蓝色） |
| 标记不处理 | 用户标记为不处理的数量（灰色） |

##### 2.2.2 筛选与列表

筛选条件（pill 形切换按钮）：

| 选项 | 筛选逻辑 |
|------|---------|
| 全部 | 显示所有已处理记录 |
| 我处理的 | 仅显示用户手动处理的记录 |
| AI 自动处理 | 仅显示 AI 自动处理的记录 |

列表项展示：图标 + 标题 + 描述 + 时间


#### 2.3 Tab 3：自动化规则

采用两栏布局：左侧为规则列表，右侧为 Chat 面板。

##### 2.3.1 规则列表

展示所有已配置的自动化规则。每条规则卡片包含：

| 元素 | 说明 |
|------|------|
| 状态指示灯 | 绿色 = 启用，灰色 = 已暂停 |
| 规则名称 | 规则标题 + 类型标签（邮件 / 跨系统） |
| 规则描述 | 规则触发条件、执行动作和 AI 执行方式的自然语言描述（由 AI 总结 Chat 面板中的对话历史自动生成，包含 Prompt 指令） |
| 运行信息 | 触发频率、上次触发时间 / 累计触发次数 |
| 开关 | 启用/暂停规则 |
| 删除 | 删除规则（hover 显示） |

规则分为两类：

| 类型标签 | 说明 |
|---------|------|
| 邮件 | 纯邮件内的操作规则（打标签、归档、自动回复、提醒等） |
| 跨系统 | 涉及外部系统联动的规则（任务逾期催促、审批查询等） |

##### 2.3.2 Chat 面板（规则创建）

用户通过自然语言在右侧 Chat 面板中描述规则需求，AI 解析后生成规则摘要卡片，用户确认即创建。

**Chat 交互流程：**

```
用户：我要设置自动化规则
   ↓
AI：请告诉我规则的触发条件、触发频率、条件逻辑和需要执行的动作
   ↓
用户：每天 9 点归纳前一天的邮件，生成 Todo List 通过 D 发送给我
   ↓
AI：生成规则摘要卡片（名称 + 描述 + 频率 + 确认模式）
   ↓
用户：点击「确认创建」按钮
   ↓
规则创建成功，出现在左侧规则列表中
```

**规则摘要卡片包含：**

| 字段 | 说明 |
|------|------|
| 规则名称 | AI 自动命名 |
| 规则描述 | 触发条件 + 执行动作的完整描述（含 Prompt 指令） |
| 触发频率 | 实时触发 / 定时触发 / 事件触发 |
| 确认模式 | 自动执行 / 待确认后执行 |

##### 2.3.3 规则字段采集

AI 在对话中需要收集以下字段信息来生成完整规则。用户可以一次性描述，也可以通过多轮对话逐步补充。

| 字段 | 必填性 | 说明 | 用户自然语言示例 |
|------|--------|------|----------------|
| 触发条件 | 必填 | 什么情况下触发规则 | "收到出版社的邮件时" "邮件里提到合作的" |
| 执行动作 | 必填 | 触发后要做什么 | "自动打标签" "给负责人发催促邮件" "转发给王明" |
| 触发频率 | 必填 | 多久检查一次 / 什么时候执行 | "每天早上 9 点" "实时" "有逾期就触发" |
| 确认模式 | 必填 | 执行前是否需要用户确认 | "发之前让我看一下" "不用每次问我" "直接执行就行" |
| 目标系统 | 条件必填 | 涉及跨系统操作时必须明确目标系统；纯邮件操作（打标签、归档）无需指定 | "在 A-System 里创建任务" "通过 D 发给我" |
| 接收人 | 条件必填 | 执行动作涉及发送/转发时必须明确接收人；打标签、归档等无需指定 | "给 BD 部门的王明" "发给负责人" |
| AI 执行策略 | 条件必填 | 对 AI 执行动作有特殊要求时需说明；无特殊要求则 AI 按合理默认值执行 | "语气正式但不强硬" "按项目名称分类" |

**缺少信息时的 AI 追问示例：**

| 缺少字段 | 用户输入 | AI 追问 |
|---------|---------|--------|
| 执行动作 | "帮我关注出版社的邮件" | "好的，收到出版社邮件后你希望我怎么处理？比如自动打标签、归档、还是转发给某个人？" |
| 触发频率 | "帮我汇总邮件生成 Todo" | "没问题。你希望什么时候汇总？比如每天早上 9 点，还是每周一早上？" |
| 确认模式 | "任务逾期了帮我发催促邮件" | "收到。催促邮件发出前需要你确认一下，还是我直接发送？" |
| 触发条件 | "帮我自动打标签" | "可以。什么样的邮件需要打标签？比如特定发件人、包含某些关键词、还是来自某个系统？" |
| 目标系统 | "邮件里提到合作的，帮我把联系人信息转发出去" | "转发到哪个系统？是通过 D 发给同事，还是在 A-System 里创建一个跟进任务？" |
| 接收人 | "A-System 里有任务逾期了，帮我发催促邮件" | "催促邮件发给谁？是任务的负责人，还是指定某个人？" |
| 多个字段 | "设置一个自动化规则" | "好的，告诉我：什么情况下触发？触发后做什么？我来帮你生成规则。" |


#### 2.4 Chat 页面对接

邮件助手中所有「确认」和「修改」操作均跳转至 chat.html，通过 URL 参数传递内容和交互模式。

**URL 参数：**

| 参数 | 值 | 说明 |
|------|---|------|
| mode | `confirm` | 内容作为用户第一条消息直接发送，AI 自动回复确认 |
| mode | `edit` | 内容回填至输入框，AI 提示用户编辑后发送 |
| content | URL encoded 文本 | 行动项的结构化内容 |

**chat.html 行为：**

| 模式 | 渲染行为 |
|------|---------|
| confirm | 清空默认会话 → 显示用户消息（已发送状态）→ AI 总结操作内容并询问「是否确认执行？」→ 显示「确认执行」按钮 → 用户点击后 AI 回复「已确认，正在执行…」 |
| edit | 清空默认会话 → AI 提示「请编辑下方内容后发送」→ 内容预填至输入框 → 等待用户修改并手动发送 |

## 四、跨系统集成

邮件助手与以下系统进行联动：

| 系统标识 | 系统名称 | 联动能力 |
|---------|---------|---------|
| @邮件 | 邮件系统 | 发送/回复/转发邮件、打标签、归档 |
| @A | A-System（任务管理） | 创建任务、查询任务状态、检测逾期 |
| @P | P-System（人员架构） | 查询员工信息、查询审批上级、权限验证 |
| @D | D-System（通讯系统） | 发送消息、转发联系人信息 |
| @日历 | 日历系统 | 创建提醒、创建日程 |

**联动方向：**

| 方向 | 说明 | 示例 |
|------|------|------|
| 邮件 → 外部系统 | AI 分析邮件内容后主动操作外部系统 | 收到合作邮件 → 提取联系人信息转发至 D-System |
| 外部系统 → 邮件 | 外部系统事件触发邮件操作 | A-System 任务逾期 → 自动生成催促邮件草稿 |
