## 一、背景

### 1. 现状

翻译官的日常工作围绕多个系统展开，每天需要在 A 系统、i18n 系统之间来回切换，手动查找任务、手动建票、手动跟进进度。具体工作内容如下：

| # | 任务类型 | 当前做法 |
|---|---------|---------|
| 1 | Text / Announcement Translation | 在 A 系统中找票，按顺序做。工作内容：AI 翻译内容监修 → 测试环境 QA → 上线 |
| 2 | Image Translation | 手动检查 i18n 中是否有图片翻译需求，有的话手动建 A 票 |
| 3 | 字典维护 | 新游戏项目：维护术语表（日语 → 英语/繁体中文/韩语）手工加入 i18n 字典；运营期：边做翻译边维护 |
| 4 | LQA | 游戏上线初期或新活动时进行走查（手动创建 A 票） |
| 5 | 给 CP 建 A 票并 review | 发现翻译相关问题（如图片文本超框）后建 A 票给 CP 修改，CP 完成后翻译官 review（通过 → Completed；不通过 → Goback） |

**关于工作时间**：完全取决于翻译任务中的翻译条数，没有固定时间。

**关于 i18n Knowledge**：调研中翻译官并没有用起来。

### 2. 翻译官的 A 票类型

翻译官日常涉及的 A 票分为三种类型：

| 类型 | 创建方 | 关闭方 | 说明 |
|------|--------|--------|------|
| 翻译任务票 | AI / 系统自动创建 | 系统自动关闭 | Text Translation、Announcement Translation、Image Translation 的翻译任务 |
| 自建走查票 | 翻译官自己创建 | 翻译官自己关闭 | 例如游戏上线后的体验走查、LQA 等 |
| 给 CP 的票 | 翻译官创建，指派给 CP | CP 完成 + 翻译官 review 后关闭 | 图片问题、走查发现的 bug 等 |

**D 系统如何区分三种票**：

根据 A 票的**指派部门**和**标题**进行自动分类：

| 判断条件 | 票类型 |
|---------|--------|
| 指派部门 = **CP** | → 给 CP 的票（Follow） |
| 指派部门 = **Game Ops** 且 标题包含 `Translation` | → 翻译任务票 |
| 指派部门 = **Game Ops** 且 标题不包含 `Translation` | → 自建走查票（Other Tasks） |

**给 CP 的票状态流转**：

```
Pending → InProgress → InProgress: review → Completed
                  ↑                    │
                  └──── Goback ────────┘
```

| 状态 | 含义 |
|------|------|
| `Pending` | 翻译官已建票，CP 尚未开始处理 |
| `InProgress` | CP 正在处理中 |
| `InProgress: review` | CP 已完成修改，等待翻译官 review |
| `Completed` | 翻译官 review 通过，票关闭 |

> 翻译官 review 不通过时，票的状态 Goback 到 `InProgress`，CP 重新修改。

### 3. 痛点

- 翻译官每天要在多个系统间手动查找和跟踪任务，效率低
- 任务优先级靠个人经验判断，没有系统化支持
- Image Translation 需要手动巡检 i18n 系统，容易遗漏
- 工作量没有统计，对"今天做了多少"缺乏可视化
- 给 CP 创建的 A 票需要手动跟踪 review 状态

### 4. 实现思路

基于 D 系统 3.0 的核心理念——**AI 创建任务、AI 监控进度、AI 收结果**（替代原来的人找活、人盯进度、人写日报），为翻译官设计执行者视图。

| 维度 | 旧模式（人找活） | 新模式（AI 派活） |
|------|----------------|----------------|
| 任务来源 | 翻译官自己去 A 系统 / i18n 找 | AI 自动拉取，按优先级排好推给翻译官 |
| 优先级 | 凭经验判断 | AI 按 DDL + 翻译量自动排序 |
| Image 翻译 | 手动巡检 i18n | AI 自动巡检，发现需求自动建票 |
| 进度监控 | 自己记住做到哪了 | AI 实时追踪，每小时汇总 |
| 工作总结 | 自己统计、手动更新 A 票 | AI 自动结算，自动更新 A 票状态 |

---

## 二、目标

1. 翻译官每天花在"找活"和"系统间切换"上的时间降为 0
2. 任务遗漏率下降（尤其是 Image Translation 的人工巡检遗漏）
3. 翻译官可聚焦翻译监修本身，工作效率提升
4. 工作量可量化，为后续预估准确度优化提供数据基础

| Image Translation 拉取 | 见下方 2.1.7 Image Translation A 票管理 |

##### 2.1.7 Image Translation A 票管理

由于 Image Translation 需求是持续新增的（i18n 中随时可能出现新的图片翻译需求），且目前没有系统能自动判断图片翻译是否完成，因此单独设计 A 票创建和结算逻辑。

**A 票创建规则**：

| 需求项 | 说明 |
|--------|------|
| 粒度 | **每个游戏一张 A 票**（同一天内同一游戏不重复建票） |
| 检测时间 | 每天 **09:00、13:00、16:00** 查询 i18n 系统 |
| 创建逻辑 | 检测到游戏有新的图片翻译需求时：若该游戏当天已有未关闭的 Image Translation A 票 → 将新增图片追加到现有票中（更新翻译量）；若该游戏当天没有 A 票 → 创建新票 |
| 推送通知 | 创建新票或追加图片后，推送通知给翻译官 |

**自动创建 A 票的字段**：

| 字段 | 值 |
|------|-----|
| 部门 | Game Ops |
| 项目 | 游戏名称（如 doraemon、kusuriya 等） |
| 负责人 | 该游戏对应的翻译官 |
| DDL | 当天 |
| 标题 | `【{游戏appid}】Image Translation` |

> 标题中包含 `Translation`，因此会被 D 系统自动分类为"翻译任务票"。

**A 票结算规则**：

| 需求项 | 说明 |
|--------|------|
| 结算时间 | 每天 **18:00** |
| 结算逻辑 | 统计该游戏当天已 approve 的图片数量，所有 Image Translation A 票统一标记为 `completed`，在 A 票的 output 字段中写入"今天 approve 了 X 张图片" |
| 未完成处理 | 如果有图片当天未 approve，记录在明日的晨间播报中提示翻译官 |

**设计原因**：

- Image Translation 与 Text/Announcement 不同，翻译工作（如修图）可能在外部工具完成，目前没有系统信号能自动判断单张图片是否完成
- 采用"每天 18:00 统一结算"的方式，避免票一直处于未关闭状态
- 每个游戏一张票，避免票数碎片化，同时清晰反映每个游戏当天的图片翻译工作量

##### 2.1.2 任务优先级排序

| 需求项 | 说明 |
|--------|------|
| 排序规则 | 第一优先级：DDL 越近越靠前；第二优先级：同一 DDL 下翻译条数越多越靠前 |
| 动态调整 | 新任务进入时，按规则自动插入到正确位置 |

##### 2.1.3 任务聚合展示

任务按翻译类型（Text Translation / Announcement Translation / Image Translation）聚合为分组卡片。每个分组卡片显示：

| 信息项 | 说明 |
|--------|------|
| 类型名称 | Text Translation / Announcement Translation / Image Translation |
| 任务数 | 该类型下的任务总数 |
| 总翻译量 | 该类型下所有任务的翻译量合计（items / Announcements / images） |
| 预估总耗时 | 该类型下所有任务的预估时间合计 |
| 状态分布 | done / in-progress / pending 各多少个 |

> 用户 hover 分组卡片时展开，显示该类型下的逐条任务卡片。

##### 2.1.4 任务卡片

每条任务卡片显示以下信息：

| 字段 | 说明 |
|------|------|
| 游戏名称 | 所属游戏（如 kusuriya、doraemon 等） |
| 翻译类型 | Text Translation / Announcement Translation / Image Translation |
| 翻译量 | 条数 / 篇数 / 张数 |
| DDL | 截止时间 |
| Est. | 预估耗时 |
| Status | pending / in-progress / done |

##### 2.1.5 任务状态定义

| 状态 | 含义 | 说明 |
|------|------|------|
| `pending` | AI 已完成翻译，待监修 | AI 翻译完成后自动进入此状态，等待翻译官开始监修 |
| `in-progress` | 监修中 | 翻译官开始监修。判断依据：从 A 系统获取票的状态变化 |
| `done` | 监修完成 | 翻译官完成监修 |

##### 2.1.6 任务卡片颜色提醒

不做主动弹窗或消息提醒，仅通过任务卡片的**视觉颜色**让翻译官感知时间消耗情况。颜色提醒**仅对 `in-progress` 状态**的任务生效。

| 条件 | 卡片表现 |
|------|---------|
| `pending` | 默认色（无颜色提醒） |
| `in-progress`，已用 < 50% 预估时间 | 默认色 |
| `in-progress`，已用 50%-90% 预估时间 | 🟡 黄色左边框 |
| `in-progress`，已用 > 90% 预估时间 | 🔴 红色左边框 |
| `done` | 降低透明度 + 标题划线 |

> `in-progress` 起始时间的判断依据：从 A 系统获取票的状态变为 `in-progress` 的时间戳。

> 今天有 11 个翻译任务需要处理，预估总耗时约 10h10min。
> 任务已按 DDL + 翻译量自动排序，从上往下做即可。祝今天工作顺利！

#### 2.4 需要跟进的 A 票（Follow）

展示翻译官**给 CP 创建的 A 票**，让翻译官实时了解 CP 的处理进度和自己需要 review 的票。

##### 2.4.1 卡片信息

| 字段 | 说明 |
|------|------|
| Category | 票的分类（Bug / General） |
| App ID | 所属游戏 |
| Title | 问题标题 |
| Context | 问题描述 |
| Severity | 严重程度 |
| Assignee | 负责人姓名 + 角色（CP） |
| Ticket Status | 票的当前状态（见下方状态展示） |

##### 2.4.2 票状态展示

每张卡片顶部以醒目的**状态横幅**展示当前状态，不同状态有不同的视觉效果：

| Ticket Status | 横幅颜色 | 卡片左边框 | 附加表现 |
|---------------|---------|-----------|---------|
| `Pending` | 灰色横幅 | 灰色 | — |
| `InProgress` | 蓝色横幅 | 蓝色 | — |
| `InProgress: review` | 橙色横幅（呼吸动画） | 橙色 | 显示 CP 留言（如有） |
| `Completed` | 绿色横幅 | 绿色 | 卡片降低透明度 + 标题划线 |

##### 2.4.3 CP 留言

当票状态为 `InProgress: review` 时，卡片中显示 CP 的修改说明，帮助翻译官了解 CP 做了什么，便于去 A 系统 review。

##### 2.4.4 操作方式

卡片可点击，点击后跳转到 A 系统对应的票进行操作（Approve → Completed / Goback）。D 系统中**不做操作按钮**，所有操作在 A 系统完成。

##### 2.4.5 一天中的状态变化

Follow A 票在不同时段展示不同的状态，体现 review 生命周期：

| 时段 | 示例 |
|------|------|
| 晨间播报 | 票 A: `Pending`（刚建票）、票 B: `InProgress`（CP 处理中） |
| 工作进行中 | 票 A: `InProgress: review`（CP 完成，需 review） + CP 留言、票 B: `InProgress` |
| 下班总结 | 票 A: `Completed`（review 通过）、票 B: `InProgress: review`（CP 刚完成，明天 review） |

#### 2.5 下班总结

AI 在下班前自动汇总当天工作成果。

| 需求项 | 说明 |
|--------|------|
| 触发时机 | 每日 17:30（Image Translation A 票结算在 18:00） |
| 推送对象 | 翻译官 + 翻译组长（决策者视图） |

**汇总统计卡片**：

| 统计项 | 说明 |
|--------|------|
| A-tickets | 今日 A 票完成数 / 总数（百分比）。包含所有类型的 A 票（翻译任务票 + 自建走查票 + Image Translation 票） |

**按翻译类型汇总**：

| 维度 | 说明 |
|--------|------|
| Text Translation | 今日翻译的 text 条数 |
| Announcement Translation | 今日翻译的 announcement 数量 |
| Image Translation | 今日 approve 的图片数量 |

**结算内容**：

- 任务卡片（按类型聚合，显示每条任务的完成状态）
- 自建走查票状态汇总
- Follow A 票的最终状态

#### 2.6 其他任务（自建走查票）

翻译官自己创建的 A 票（LQA 走查、游戏体验等），由翻译官自行管理和关闭。D 系统对这类票做**展示和汇总**，不做自动化处理。

| 需求项 | 说明 |
|--------|------|
| 数据来源 | A 系统中翻译官自己创建、且指派给自己的票 |
| 展示位置 | Worker 视图 Dashboard 右侧，独立模块"其他任务（Other Tasks）" |
| 卡片信息 | 游戏名称、标题、状态（pending / in-progress / done）、创建时间 |
| 操作 | 翻译官在 A 系统中手动关闭，D 系统同步状态变更 |
| 结算 | 下班总结中统计当天自建票的完成数量 |

> 这类票的创建和关闭完全由翻译官控制，D 系统只做被动同步和展示。

| Image Translation | 15 min/image | 涉及图片中的文字提取和排版 |

## 五、数据源与接口

### 1. A 系统

| 接口 | 说明 |
|------|------|
| 获取翻译票列表 | 拉取指定翻译官的所有翻译票（Text / Announcement / Image） |
| 票状态变更监听 | 监听票状态变化（Pending → InProgress → InProgress: review → Completed），用于更新 Follow A 票展示 |
| 票状态更新 | AI 结算时自动关闭已完成的票、更新未完成票的进度 |

### 2. i18n 系统

| 接口 | 说明 |
|------|------|
| Image Translation 巡检 | 每小时检查所有游戏的 Image Translation 模块，对比上次巡检结果发现新增项 |

## 六、暂不纳入的功能

| 功能 | 原因 |
|------|------|
| 字典维护——AI 自动发现未收录术语 | 需要实现翻译文本与字典的实时比对和专有名词提取，当前暂无法实现 |
| LQA 提醒——AI 自动感知游戏活动上线 | 活动信息来源分散（钉钉群、运营排期、i18n 数据），且不是所有游戏都有结构化的活动数据，实现不现实 |
| Image Translation 单张图片完成检测 | 目前 i18n 系统没有"已翻译/未翻译"状态字段，无法自动判断单张图片是否完成。采用每日 18:00 统一结算的方式替代 |

## 七、设计决策待确认

| # | 问题 | 选项 | 建议 |
|---|------|------|------|
| 1 | Image Translation 自动建票的权限 | A) AI 自动建票 B) AI 发现后推送给翻译官确认再建 | 先 B，确认效果稳定后再切 A |
| 2 | 预估耗时系数 | 需要和翻译团队一起定每种翻译类型的平均耗时 | 安排一次沟通确认 |
