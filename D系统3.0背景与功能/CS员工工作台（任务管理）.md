# 背景

当前业务数字化程度很高，通过员工手动创建任务进行管理过于低效，转换为AIAgent创建任务并跟进。

# 目标

* CS 岗位的任务创建和结算由 Agent 自动完成
* 后续一线岗位的任务管理模式以此为模板进行迁移

## AI 输出策略

本模块定义 AI Agent 在 Worker 视图中的全部输出行为。AI 并非被动响应用户提问，而是**主动监测员工工作状态并在关键节点推送结构化信息**。


1. 输出触发机制：AI Agent 基于以下四种场景主动生成输出内容
   * 规则：
     * **早晨任务播报**：每个工作日开始时（==员工本地时区上班时间==）自动触发，输出当日任务概览
     * **无操作督促**：检测到员工连续 30 分钟无任何操作后触发，输出工作状况确认和紧急任务提醒
     * **任务全部完成**：所有当日任务的待处理数量归零时触发，输出完成庆祝和数据汇总
     * **下班前积压告警**：距离下班时间 2 小时且仍有未完成任务时触发，输出积压分析和优先级建议
   * 边界：
     * 同一场景在同一工作日内不重复触发（例如 30 分钟督促如果用户已恢复操作又再次无操作，可再次触发，但完成庆祝只触发一次）
     * 非工作日和非工作时段不触发督促类输出
     * 所有触发时间基于员工所在地时区计算
2. 输出内容结构：AI 输出由三种内容元素组合而成
   * 规则：
     * **文本段落**：普通文字信息，以逐字打字机效果呈现。当前正在输出的行为深色高亮字体，已输出完成的行自动变为浅灰色
     * **任务卡片组**：以卡片列表形式展示任务，每张卡片包含任务图标、任务名称、任务描述和待处理数量。卡片可点击跳转到对应工作台
     * **任务引用标记**：在后续详细说明中引用某个任务时，以小图标+任务名称的形式出现，起到段落分隔和语境切换的作用
   * 状态：
     * 文本行：正在输出（深色 + 光标闪烁）→ 输出完成（浅灰色）
     * 任务卡片：未完成状态（显示待处理数量）→ 全部完成状态（显示完成勾选标记，整体变为半透明）
   * 边界：
     * 空文本行作为段落间距使用，不显示任何字符
     * 任务卡片的待处理数量从各对应业务系统实时获取
3. 输出渲染策略：流式逐步呈现，模拟 AI 实时思考的效果
   * 规则：
     * 内容按定义顺序逐项输出，当前项输出完成后自动开始下一项
     * 文本行以打字机效果逐字显示，速度为每字符 28ms
     * 空文本行（段落间距）停留 200ms 后自动推进
     * 非空文本行输出完成后停留 600ms 再推进到下一项
     * 任务卡片组以淡入+上滑动画逐张出现，每张间隔 150ms，整组停留 1800ms 后推进
     * 任务引用标记以滑入动画出现，停留 300ms 后推进
   * 交互：
     * 内容区域支持纵向滚动，新内容输出时自动滚动到底部
     * 任务卡片支持 hover 效果（略微上浮 + 背景加深），点击在新窗口打开对应工作台
   * 边界：
     * 全部内容输出完成后，页面保持静止，不再有任何动画
4. 各场景输出模板：

   **场景一：早晨任务播报**
   * 文案：问候语（"早上好 ☀️"）→ 任务数量说明 → 任务卡片组（全部当日任务）
   * 规则：任务卡片按优先级排序，每张卡片显示该任务类型当前的待处理数量

   **场景二：30 分钟无操作督促**
   * 文案：无操作检测说明 → 工作状况确认 → 紧急任务卡片组（仅未完成任务）→ 逐任务展开说明（待处理数量、当前进度）→ 帮助说明 → 恢复工作提示
   * 规则：仅展示有待处理项的任务，已完成的任务不在督促中展示；逐任务说明需包含具体待处理数据

   **场景三：任务全部完成**
   * 文案：庆祝语 → 完成声明 → 任务卡片组（全部显示已完成状态）→ 今日完成情况总结 → 逐任务完成数据 → 持续关注提示 → 表扬鼓励
   * 规则：任务卡片全部显示为完成态（待处理数为 0，显示勾选标记）；完成数据包含处理数量、关键指标（如平均响应时间）

   **场景四：下班前任务积压**
   * 文案：时间告警 → 积压检测说明 → 任务卡片组（显示积压数量）→ 逐任务积压详情 → 建议处理策略（优先级排序）→ 延期/转交协助说明 → 鼓励
   * 规则：任务卡片显示当前积压数量（可能高于早晨的待处理数，因为日间持续有新任务进入）；建议策略按紧急程度排序
5. 任务积压判断策略：系统如何判定当前存在"任务积压"并触发告警
   * 规则：
     * 积压判定**在下班前 2 小时**执行一次检查（基于员工本地时区的下班时间）
     * 判定逻辑综合以下三个维度，**任一维度命中即判定为积（根据实现难度先高二、三）**：
       * **维度一：剩余量 vs 处理能力**
         * 计算员工当日各任务类型的**平均处理速率**（= 当日已完成数 ÷ 已工作时长）
         * 计算按当前速率清空全部剩余待处理项所需的**预估时间**
         * 如果预估时间 > 剩余工作时间（即下班前无法清空），判定为积压
         * 示例：已工作 6 小时处理了 18 个工单（速率 3 个/小时），剩余 23 个待处理，需 7.7 小时，但只剩 2 小时 → 积压
       * **维度二：完成率低于阈值**
         * 当日总待处理量 = 早晨初始待处理 + 日间新增
         * 当日已完成量 = 各任务类型已完成数之和
         * 如果整体完成率（已完成 / 总待处理）< **50%** 且距离下班 ≤ 2 小时，判定为积压
       * **维度三：单任务零进展**
         * 如果某个任务类型的待处理数量 > 0 且当日已完成数 = 0（即完全没有开始），判定为积压
         * 此维度用于捕捉"部分任务被遗忘"的情况
   * 边界：
     * ==如果员工当日请假半天（如下午才上班），处理速率和剩余时间按实际在岗时段计算——关联 P==
     * 如果所有任务的待处理数均为 0，不触发积压检查（已进入"任务全部完成"场景）

## 客户端通知

本模块定义 AI 输出内容如何通过客户端通知触达用户。通知不是摘要或模板化消息，而是**直接弹出一个窗口，窗口内承载完整的 AI 输出内容**，与 Worker 视图中的流式输出体验一致。


1. 通知形式：桌面弹出窗口 + 提示音
   * 规则：
     * 每次 AI 生成输出内容时，在员工桌面**直接弹出一个独立窗口**
     * 窗口内容即为 AI 输出的完整内容（文本段落、任务卡片组、任务引用标记），以与 Worker 视图相同的打字机流式效果实时呈现
     * 弹出窗口的同时播放**提示音**，确保员工在专注其他工作时也能感知到通知
     * 不使用通知摘要或通知模板，窗口本身就是 AI 的完整输出界面
   * 交互：
     * 弹出窗口可拖拽移动、可调整大小
     * 窗口内的任务卡片可直接点击，跳转到对应工作台
     * 窗口右上角有关闭按钮，关闭后内容仍保留在 Worker 视图中可随时回看
     * 如果员工未关闭窗口，新的 AI 输出将在同一窗口中追加（而非弹出多个窗口）
   * 状态：
     * 弹出态：窗口出现在桌面前台，AI 内容正在流式输出
     * 输出完成态：内容输出结束，窗口保持打开，员工可阅读和操作
     * 已关闭态：员工手动关闭窗口，内容归档到 Worker 视图
   * 边界：
     * 员工已在 Worker 视图且页面处于前台可见时，不弹出独立窗口（内容直接在页面内展示即可）
     * 同一时间最多只有一个弹出窗口，不会出现多窗口叠加
     * 如果浏览器不支持弹出窗口（被拦截），降级为浏览器原生 Notification API 推送，点击后打开 Worker 视图
2. 提示音策略：
   * 规则：
     * 每次弹出窗口时播放一次提示音，AI 输出过程中不重复播放
     * 提示音为简短、轻柔的通知音效，不超过 2 秒
   * 优先级与音效映射：
     * 早晨任务播报 → 标准提示音
     * 无操作督促 → 标准提示音（连续播放 2 次，间隔 1 秒）
     * 任务全部完成 → 愉悦的完成音效
     * 下班前积压告警 → 紧急提示音（音调略高）
   * 边界：
     * 员工可在个人设置中关闭提示音（静音模式），此时仅弹出窗口不播放声音
     * 凌晨结算通知（00:00）不弹出窗口、不播放提示音，仅静默写入 Worker 视图，员工次日上班时自然看到

## A 票与每日任务管理

本模块定义每日任务如何自动创建为 A 票，以及 A 票生命周期中的自动化行为。


1. A 票自动创建：每个工作日自动为员工创建当日的 A 票
   * 规则：
     * 创建时机：员工本地时区的工作日开始时（上班时间），系统自动创建
     * 根据客服员工 skill ，每天创建 1-3 张A 票，该员工当日每种类型的任务汇总为一张票
     * A 票标题格式：客服工单处理、知识库维护、服务评估
   * 边界：
     * 非工作日不创建 A 票（除非员工被安排值班）
     * 如果员工当日请假，结算时注明请假时长
     * A 票创建失败时通知管理员，通知研发进行告警
2. A 票子任务结构：
   * 
| 子任务 | 量化指标 | 数据来源 |
|-----|------|------|
| 客服工单处理 | 当日处理工单数量（Expert 回复 AI j 个，CS 系统直接回复 k 个） | 工单中心 |
| 知识库维护 | 知识库改善任务条目数（冲突任务x 个，重复知识任务 y 个，低热度知识 z 个） | Knowledge Base |
| 服务评估 | 评估对话组数 | CS Quality Panel |
   * 边界：
     * 子任务类型由员工 Skill 决定，比如有的不负责评估、改善
     * 如果对应业务系统不可用，给研发发出告警

## Output 自动结算

本模块定义每日结束时系统自动为 A 票填写 Output 的策略，包括结算时机、数据采集和 Output 格式。


1. 结算时机：基于员工所在地时区执行
   * 规则：
     * 每天当地时间 **00:00**（午夜零点）自动触发结算
     * 系统根据员工档案中记录的时区字段确定触发时间（如：东京员工 UTC+9 = UTC 15:00 触发；旧金山员工 UTC-8 = UTC 08:00 触发）
     * 结算窗口为当地时间 `00:00:00` \~ `00:05:00`，在此窗口内完成数据采集和 Output 写入
   * 边界：
     * 如果员工时区信息缺失，默认使用JST
     * 如果结算时数据源不可用，给研发预警
     * 美国夏令时切换日按新时区执行
2. 数据采集：从各业务系统收集员工当日的实际完成数据
   * 规则：
     * 采集时间范围：员工当地时间 `00:00:00` \~ `23:59:59`（即完整的一个自然日）
     * 按子任务类型分别采集对应指标：
     * 所有指标均从业务系统 API 实时拉取，不依赖客户端上报
   * 边界：
     * 跨时区协作场景下，以员工本人时区为准（如东京员工和旧金山同事协作处理同一工单，各自按自己的时区结算）
     * ==工单处理时间横跨多天的，归属于工单关闭时所在的那个自然日==
3. 格式

   **==A 票 Output 格式==**==（新结构）：==

   ==■ 客服工单处理==

     ==处理工单总数：{N} 个==

     ==· Expert 回复 AI 处理：{j} 个==

     ==· CS 系统直接回复：{k} 个==

   ==■ 知识库维护==

     ==知识库改善任务条目数：{N} 条==

     ==· 冲突任务：{x} 个==

     ==· 重复知识任务：{y} 个==

     ==· 低热度知识：{z} 个==

   ==■ 服务评估==

     ==评估对话组数：{N} 组==
