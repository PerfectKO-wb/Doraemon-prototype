## 一、背景

### 1. 现状

当前游戏多语言翻译的协作链路中，日语翻译专家在完成对游戏日语的二次监修后，需要手动触发通知，告知其他语种的翻译专家进行后续监修。


| 角色     | 载体                     | 说明                      |
| ------ | ---------------------- | ----------------------- |
| 日语专家   | i18n text table（翻译工作台） | 完成游戏日语的二次监修工作           |
| 其他语种专家 | 消息中心                   | 接收日语专家发出的监修通知，并在完成后进行确认 |


**当前信息流转路径**：

```text
日语专家完成日语二次监修
    │
    ▼
日语专家手动点击 "Send Notification" 按钮  ⚠️ 容易遗漏
    │
    ▼
系统向其他语种专家发送通知
    │
    ▼
其他语种专家在【消息中心】收到通知  ⚠️ 不明显，易被忽略
    │
    ▼
其他语种专家完成监修并确认
```

### 2. 痛点

- **漏发通知导致流程阻塞**：日语专家在完成监修后，经常会忘记手动点击 `Send Notification` 按钮，导致下游其他语种的监修工作停滞，影响整体进度。
- **通知触达率低**：通知发送至其他语种专家的"消息中心"，该渠道信息繁杂且提醒不明显，专家容易漏看或延迟查看。
- **确认链路长**：其他语种专家缺乏便捷的确认入口，无法在沟通的上下文中直接完成工作交接。

### 3. 实现思路

> **为何不采用自动触发方案**：系统无法准确判断日语专家是否已完成监修——专家可能分多次编辑、中途暂停或仅修改部分词条，不存在明确的"监修完成"状态节点可供自动触发。

因此采用 **通知渠道优化** 的方案：

1. 移除工具栏的通知按钮（喇叭图标）。
2. 将通知渠道从"消息中心"迁移至高频使用的 **i18n 咨询会话**中，通过 `@对应人员` 强提醒，并提供 `mark as reviewed` 快捷操作按钮。
3. 结合 Deploy 操作，在 Deploy 时一并下发 i18n 任务。

**核心设计原则**：

- **基于会话的协作 (ChatOps)**：在 i18n 咨询会话中通过 `@对应人员` 强提醒，并在会话消息中嵌入 `mark as reviewed` 按钮，实现"通知-处理-确认"的闭环。

---

## 二、目标

1、**降低漏发风险**：将通知逻辑与 Deploy 绑定，减少专家忘记发送通知的情况。
2、**提升响应时效**：通过会话内 `@` 强提醒，提高其他语种专家的消息触达率和响应速度。
3、**优化协作体验**：提供会话内的 `mark as reviewed` 快捷操作，缩短专家确认路径，提升工作效率。

---

## 三、需求

### 1. 功能范围

- **提醒对象**：日语翻译专家（发送通知的操作人）。
- **发送载体**：i18n text table 工具栏中的 Deploy 按钮。
- **通知目标**：对应的 i18n 咨询会话（i18n consultation session）。
- **通知形式**：系统在会话中发送包含 `@对应语种专家` 的提醒消息，并附带 `mark as reviewed` 交互按钮。

### 2. 功能需求清单

#### 2.1 Deploy 弹窗与下发任务

| 需求项   | 说明                                                                                           |
| ----- | -------------------------------------------------------------------------------------------- |
| 触发方式  | 点击工具栏右侧的 Deploy 按钮（位置：`.deploy-top-btn`），唤起 Deploy 弹窗。                                                         |
| 弹窗字段  | 按顺序展示：**Game**（展示当前游戏，例：`gamedemo`）、**Language**（展示目标语言，固定显示 `JA`）、**Submission ID**（条件显示，见下行）、底部主按钮 `Deploy & Notify`。 |
| SubmissionID 显示 | 如果用户在工具栏左侧的 **"All submissions" 筛选器**中选定了特定的 submissionID，则 Deploy 弹窗内展示对应的 **Submission ID** 字段（只读，展示完整 ID），且 "All submissions" 按钮文本会同步更新为所选 ID 的截断形式（例：`submission-d3387477...`）。未选定时弹窗内不展示该字段，按钮保持 `All submissions`。 |
| 按钮文案动态切换 | 主按钮文案随当前视图中是否存在 JA 变更条目（即 `.ja-cell` 中 New/Old 对比行）动态切换：<br>• **存在 JA 变更** → 按钮显示 `Deploy & Notify`；<br>• **无 JA 变更** → 按钮显示 `Deploy to Production`（即仅部署、不触发通知）。 |
| 发送反馈  | 点击主按钮后无二次确认，直接执行并关闭弹窗，在页面顶部显示 Toast：**"Deployed and notification sent successfully"**。 |
| 发送后处理 | 发送成功后：1) 当前视图中的 JA 变更对比态（New/Old）收敛为平铺文本，变更状态标记为"已通知"；2) 系统为本次发送的词条生成（或复用所选的）唯一 **SubmissionID**，绑定到会话消息中下发；3) 在对应的 i18n 咨询会话中发送通知消息。 |


#### 2.2 会话内通知与强提醒 (@)


| 需求项    | 说明                                                                           |
| ------ | ---------------------------------------------------------------------------- |
| 消息触达渠道 | 在对应的 **i18n 咨询会话** 中由系统账号（`Xiaobao` / `From: system`）发送通知消息。                                                  |
| @ 提醒机制 | 消息顶部以 `@陳柏全`、`@林 幹夫` 等形式精准 `@` 到所有需监修的其他语种专家，触发系统强提醒。                              |
| 消息卡片结构 | 会话消息以卡片形式承载任务信息，包含以下内容块：<br>• 顶部再次 `@` 对应专家；<br>• 🚩 标题行 `Text Translation Update`；<br>• 基础信息列表：`Game`、`Task Type`（如 `text`）、`Task ID`（如 `text-submission-bff71cdc-439f-44ca-bf72-5ee7e334aca2`，作为本次下发的 SubmissionID）；<br>• `Translation Requirements` 子列表：按翻译方向逐条列出（如 `JA → EN`、`JA → ZH-TW`、`JA → KO`），每条关联对应 `@专家` 及一个 **i18n Task Table** 外链跳转按钮；<br>• AI 备注行（如"日文翻译已修改 请完成从 ja 到 en 的翻译审核"）；<br>• 底部 `Mark as Reviewed` 交互按钮（见 2.3）。 |
| 消息中心跳转 | 保留原有"消息中心"的通知消息，但点击消息中心中的该通知后，不再展示原有详情页，而是**直接跳转到对应的 i18n 咨询会话**中，定位到相关通知消息。 |
| 会话筛选辅助 | 专家侧会话列表提供 `@me` 筛选 chip（`#filterAtMeBtn`），快速筛出 `@` 到自己的会话，降低漏看概率。 |


#### 2.3 会话内快捷确认 (Mark as Reviewed)


| 需求项    | 说明                                                            |
| ------ | ------------------------------------------------------------- |
| 交互按钮   | 在通知消息卡片底部，提供一个 `Mark as Reviewed` 按钮（`#markReviewedBtn`）。              |
| 权限控制   | 仅允许被 `@` 的对应语种专家点击生效，非相关人员点击无效（**需求项，待后端接入身份判断**；当前原型未做身份拦截，任何人点击都会触发按钮状态变更）。 |
| 状态同步   | 专家在会话中点击 `Mark as Reviewed` 后，系统需自动将该专家对应语种方向的监修状态同步回翻译管理系统，标记为已完成。 |
| 按钮状态变更 | 点击后按钮实时更新为 **`Deployment Successful`** 并置为禁用态（`disabled`），避免重复点击。 |

#### 2.4 会话入口与 SubmissionID 筛选

| 需求项 | 说明 |
| ----- | ----- |
| SubmissionID 生成 | 复用系统现有的 **Update 状态标记逻辑**来判断哪些词条需要被 check。日语专家点击 `Deploy & Notify` 发送通知时，系统会自动筛选出当前被标记为 Update 状态的 JA 词条，为这些数据生成一个唯一的 SubmissionID，并将该 SubmissionID 与会话消息绑定后下发。 |
| 会话入口 | 其他语种专家（EN、ZH-TW、KO）收到会话中的 `@` 通知后，点击消息进入 i18n text table。 |
| 自动筛选 | 进入 i18n text table 时，系统根据消息中的 SubmissionID **自动筛选**出本次需要监修的词条：JA → EN、JA → ZH-TW、JA → KO 三条翻译方向的词条，并展示在表格中。 |
| 筛选范围 | 每个语种专家只看到与其负责方向相关的词条。例如 EN 专家只看到 JA → EN 的词条，不看到 JA → KO 的词条。 |
| 状态关联 | 专家在 i18n text table 中完成某方向的监修（点击 Mark as Reviewed）后，对应词条的该方向状态同步更新为"已完成"。 |

#### 2.5 退出提醒弹窗（Leave Warning）

| 需求项 | 说明 |
| ----- | ----- |
| 触发条件 | 日语翻译专家在 i18n text table 中存在未发送的 JA 变更（即 `.ja-cell` 计数 > 0）时，以下离开行为会触发提醒：<br>• 点击标题栏左侧的**返回按钮**（`.back-btn`）；<br>• 点击左侧菜单栏任意一级项目（`.sidebar-nav-item`）或其子链接（`.sidebar-sub-link`）。<br>*浏览器关闭/刷新的拦截暂未接入，作为后续扩展项保留。* |
| 弹窗文案（英文） | 标题：**"You have modified Japanese translations."**<br>正文：**"Do you want to send notifications to translators?"** |
| 弹窗按钮 | 两个按钮：**"Exit"**（关闭弹窗，直接执行原导航动作）和 **"Send"**（关闭弹窗并直接唤起 Deploy 弹窗）。 |
| 行为 | 点击 "Exit"：关闭 Leave Warning，按原计划退出/跳转，不发送通知。<br>点击 "Send"：关闭 Leave Warning 并立即打开 Deploy 弹窗（2.1），用户可执行 `Deploy & Notify` 完成通知发送。 |
| 不触发条件 | 当前视图内无任何 JA 变更（`.ja-cell` 计数为 0）时，所有上述离开行为均直接放行，**不弹出**提醒。 |

## 四、附录

### 流程图（优化后）

```text
日语专家在 i18n text table 中修改日语翻译（JA 列出现 New/Old 对比态）
    │
    ▼
日语专家点击工具栏右侧 Deploy 按钮 → 弹出 Deploy 弹窗
（字段：Game / Language / Submission ID[条件] / 主按钮）
    │
    ├─ 若视图存在 JA 变更 → 主按钮文案：Deploy & Notify
    └─ 若无 JA 变更       → 主按钮文案：Deploy to Production
    │
    ▼
点击主按钮（Deploy & Notify）
    │
    ├──▶ 关闭弹窗 & Toast："Deployed and notification sent successfully"
    │
    ├──▶ JA 单元格从 New/Old 对比态收敛为平铺文本
    │
    ├──▶ 生成（或复用所选）SubmissionID，绑定本次词条
    │
    └──▶ 在 i18n 咨询会话中由 Xiaobao（system）发送消息卡片：
         @对应语种专家 · 🚩 Text Translation Update
         · Game / Task Type / Task ID(=SubmissionID)
         · Translation Requirements：JA→EN / JA→ZH-TW / JA→KO
           每条含 @专家 与 "i18n Task Table" 跳转
         · [Mark as Reviewed] 按钮
              │
              ▼
         其他语种专家点击消息（或"消息中心"的该通知）跳转到会话
              │
              ▼
         点击卡片中 "i18n Task Table"，进入 i18n text table
         并根据 SubmissionID 自动筛选出该专家方向的词条
              │
              ▼
         专家完成监修，点击卡片中的 [Mark as Reviewed]
              │
              ▼
         按钮变为 "Deployment Successful" 并禁用；
         对应语种方向状态同步为已完成
```
