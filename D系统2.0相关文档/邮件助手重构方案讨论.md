## 一、核心矛盾分析

### 1.1 D 系统 3.0 的定位变化

D 系统 3.0 要成为**"人-AI Agent-人"控制链的基础设施**，核心转变：

| 维度 | 旧定位（2.0） | 新定位（3.0） |
|------|-------------|-------------|
| 系统角色 | 通讯 & AI 对话工具 | AI 任务管理基础设施 |
| AI 定位 | 被动响应用户指令 | 主动分派、监控、督促、验收 |
| 用户关系 | 用户 → 使用工具 | AI Agent → 管理用户完成工作 |
| 组织方式 | 按功能模块（邮件、聊天、文档…） | 按角色视图（一线员工、决策者、维护者） |

### 1.2 MVP 已建立的交互范式

CS 员工工作台（Worker 视图）已经定义了一套成熟的交互范式：

- **AI 主动推送**：基于场景触发（早晨播报、无操作督促、完成庆祝、积压告警）
- **流式输出**：打字机效果、任务卡片组淡入、结构化标签协议
- **A 票驱动**：每日任务自动创建 A 票，自动结算 Output
- **单一视图**：一线员工只需关注 Worker 视图，AI 把一切信息推到面前

### 1.3 邮件助手 v1 的设计假设已不成立

| v1 假设 | 3.0 现实 | 冲突 |
|---------|---------|------|
| 邮件助手是一个独立应用入口 | 一线员工的核心入口是 Worker 视图 | 多入口分散注意力，违背"AI 把信息推到面前"的理念 |
| 用户主动进入邮件助手查看待处理 | AI 应该主动推送信息到用户面前 | 被动等用户来看 ≠ 主动触达 |
| 三 Tab 架构（待处理/已处理/规则） | Worker 视图已有任务播报、统计、督促能力 | 功能重叠，交互范式不统一 |
| 通过跳转 Chat 完成操作 | Worker 视图内直接完成交互闭环 | 多次跳转破坏效率 |
| 面向所有用户无差别 | 三类角色（一线/决策/维护）有不同需求 | 缺少角色适配 |

## 二、三个重构方案

### 方案 A：邮件能力完全融入 Worker 视图（取消独立入口）

#### 核心思路

邮件不再是一个"应用"，而是 AI Agent 的**感知通道**之一。AI Agent 持续监控邮件，将需要处理的邮件事件**转化为任务**，直接纳入 Worker 视图的任务体系。

#### 架构变化

```
旧：Workspace → 邮件助手（独立页面）→ 任务卡片 → 跳转 Chat
新：Worker 视图 ← AI Agent ← [邮件通道 + CS系统通道 + 其他通道]
```

#### 具体设计

**1）邮件 = 新的任务来源**

在 Worker 视图的 A 票体系中，邮件产生的工作变成新的任务类型：

| 现有任务类型 | 新增：邮件任务类型 |
|------------|-----------------|
| 客服工单处理 | 邮件回复处理 |
| 知识库维护 | 邮件跨系统联动 |
| 服务评估 | — |

AI Agent 分析收到的邮件后，按优先级将需要处理的邮件生成任务项，纳入现有的 `<task-cards>` 体系：

```
<task-cards>
<task id="tickets" pending="12" />
<task id="knowledge" pending="5" />
<task id="evaluation" pending="8" />
<task id="email_replies" pending="3" />     ← 新增
</task-cards>
```

**2）邮件任务在播报/督促中自然出现**

早晨播报：
```
早上好 ☀️
今天有 4 项任务，其中有 3 封邮件需要你回复。

<task-cards>
<task id="tickets" pending="12" />
<task id="email_replies" pending="3" />
<task id="knowledge" pending="5" />
<task id="evaluation" pending="8" />
</task-cards>

<task-ref id="email_replies" />
有 1 封来自田中太郎（ABC出版社）的邮件标记为紧急，建议优先处理。
```

**3）点击邮件任务卡片 → 进入邮件处理台**

点击 `email_replies` 任务卡片后，打开邮件处理台（不是独立页面，而是类似 CS 工单处理面板的并列视图），展示：

- 左侧：待回复邮件列表（带 AI 摘要和优先级排序）
- 右侧：选中邮件的详情 + AI 生成的回复草稿 + 操作按钮（发送/编辑/跳过）

**4）自动化规则 → 迁入维护者视图**

邮件自动化规则不再由一线员工自行配置，而是在**维护者视角的 works 模块**中管理，作为 Agent 的工作流之一。

#### 优缺点

| 优点 | 缺点 |
|------|------|
| 完全符合 3.0"任务管理基础设施"定位 | 邮件处理的深度被简化（原文查看、邮件线程、附件等承载力不足） |
| 用户只需关注 Worker 视图一个入口 | 对邮件重度用户（如商务、运营岗位）可能不够 |
| 与 MVP 交互范式完全一致 | 邮件任务和现有任务的粒度差异大（一封邮件 vs 一个工单） |
| 开发成本相对低，复用已有 Worker 框架 | 扩展到非 CS 岗位时可能需要重新设计邮件任务的呈现方式 |

#### 适用场景

最适合**一线员工邮件需求简单**的场景（如 CS 客服岗位，邮件主要是通知类、工单关联类），邮件只是多个任务通道之一。

### 方案 B：邮件助手作为专项任务工作台（Worker 子视图）

#### 核心思路

保留邮件助手的独立处理能力，但不再作为 Workspace 的独立应用入口，而是**从 Worker 视图中进入的专项任务工作台**——类似于点击"客服工单处理"卡片会进入 CS 系统工作台一样，点击"邮件任务"卡片进入邮件工作台。

#### 架构变化

```
旧：Workspace → 邮件助手（独立应用）
新：Worker 视图 → [邮件任务卡片] → 邮件工作台（专项视图）
```

#### 具体设计

**1）Worker 视图中的邮件任务入口**

与方案 A 相同，邮件任务作为 `<task-cards>` 中的一项出现。但点击后不是进入简单的处理面板，而是进入完整的**邮件工作台**。

**2）邮件工作台 = 重构后的邮件助手**

邮件工作台采用**与 Worker 视图一致的 AI 主动推送范式**，但聚焦邮件场景：

**进入时 AI 自动播报：**
```
你有 3 封需要处理的邮件 📬

<email-cards>
<email id="e001" from="田中太郎" subject="天剑监修第三轮反馈" priority="urgent" />
<email id="e002" from="李明" subject="Q2 预算确认" priority="normal" />
<email id="e003" from="ABC出版社" subject="合同续签事宜" priority="normal" />
</email-cards>

<email-ref id="e001" />
田中太郎发来了监修反馈，包含 5 项修改意见。
我已经生成了回复草稿，请确认后发送。
```

**3）邮件卡片的交互**

每封邮件卡片可展开，展开后显示：

| 区域 | 内容 |
|------|------|
| 邮件上下文 | 发件人、主题、时间、原文（可展开） |
| AI 分析摘要 | 一句话概括邮件内容和关键信息 |
| AI 行动方案 | 回复草稿、跨系统操作建议等（每项带确认/编辑按钮） |
| 关联任务 | 如果该邮件与现有 A 票关联，显示关联信息 |

**操作闭环在当前页面完成**（不跳转 Chat）：

- 确认回复 → 直接在工作台内发送，AI 显示"已发送"确认
- 编辑回复 → 在当前页面弹出编辑面板，修改后发送
- 跨系统操作 → 直接在当前页面确认执行，AI 显示执行结果

**4）A 票 & Output 对接**

邮件处理的结果自动记录到 A 票的 Output 中：

```
■ 邮件处理
  回复邮件总数：{N} 封
  · 确认 AI 草稿直接发送：{a} 封
  · 编辑后发送：{b} 封
  · 标记不处理：{c} 封
  跨系统联动：创建 A 任务 {x} 个
```

**5）自动化规则的归属**

与 Worker 视图中其他任务类型的自动化逻辑一致：
- 基础规则由维护者在 works 模块中配置
- 一线员工可以在邮件工作台中向 AI 提需求（如"以后这种邮件帮我自动回复"），AI 生成规则建议提交给维护者审批

#### 优缺点

| 优点 | 缺点 |
|------|------|
| 保留邮件处理的深度和完整性 | 需要新开发邮件工作台，成本比方案 A 高 |
| 与 3.0 任务框架完全兼容（从 Worker 进入，隶属于任务） | 邮件工作台和 Worker 视图的AI推送可能重复 |
| 操作闭环在同一页面，不跳转 Chat | 对不需要复杂邮件处理的岗位可能过重 |
| 邮件处理数据自然纳入 A 票结算 | 需要定义邮件工作台和 Worker 视图的数据同步机制 |
| 可以扩展到其他岗位（商务、运营等邮件重度场景） | — |

#### 适用场景

适合**邮件处理是重要工作内容**的岗位（如商务拓展、运营对接、项目管理），需要完整的邮件上下文和操作能力。

---

### 方案 C：邮件能力拆散到角色视图中（去掉"邮件助手"概念）

#### 核心思路

最彻底的重构——不再存在"邮件助手"这个概念。邮件能力被拆解为 AI Agent 的底层 Tool，根据 3.0 的三类角色，分别在各自视图中以最合适的形态呈现。

#### 架构变化

```
旧：邮件助手 = 一个完整应用
新：邮件 = Agent 的 Tool，能力分散到三个角色视图

┌─────────────────────────────────────────────┐
│                 邮件 Tool（底层能力）            │
│  收发邮件 / 分析内容 / 打标签 / 归档 / 搜索       │
└──────────┬──────────────┬──────────────┬─────┘
           │              │              │
    ┌──────▼──────┐ ┌─────▼──────┐ ┌────▼──────┐
    │  一线员工视图  │ │  决策者视图  │ │  维护者视图  │
    │  Worker     │ │  Dashboard │ │   Works   │
    └─────────────┘ └────────────┘ └───────────┘
```

#### 具体设计

**1）一线员工视图（Worker）**

邮件完全融入任务流，与方案 A 类似，但更轻量：

- AI Agent 分析邮件后，将行动项**内联到 Worker 的流式输出中**（不作为独立任务卡片）
- 邮件回复操作以**inline action**形式出现在 AI 输出文本中

```
<task-ref id="tickets" />
客服工单中有 3 个等待时间较长。
另外，田中太郎刚发来一封邮件需要回复：

📧 天剑监修第三轮反馈 — 田中太郎
AI 已生成回复草稿：「田中先生，感谢反馈，5 项修改意见已记录，预计周五前完成修订……」
[发送回复]  [编辑]  [稍后处理]
```

- 邮件信息不需要独立的工作台，直接在 Worker 视图中完成轻量操作
- 复杂邮件（需要完整阅读原文、长文编辑）点击后弹出侧边面板

**2）决策者视图（Dashboard）**

决策者看到的是**邮件相关的管理数据**：

| 数据维度 | 呈现方式 |
|---------|---------|
| 各员工邮件处理效率 | 指标卡片：平均响应时间、当日处理量 |
| 未响应邮件积压 | 告警列表：哪些员工有邮件积压、超时未回复 |
| 跨系统联动统计 | 图表：邮件触发了多少 A 票创建、多少跨部门协作 |
| AI 自动处理占比 | 趋势图：自动化率随时间变化 |

决策者不处理邮件，只看邮件作为"工作效率指标"的一面。

**3）维护者视图（Works / Tools）**

维护者在 works 模块中管理邮件相关的 Agent 工作流：

| 模块 | 邮件相关能力 |
|------|------------|
| Works（工作流） | 配置邮件监控规则、自动化流程（对应 v1 的"自动化规则" Tab） |
| Tools（工具） | 管理邮件 API 连接、配置邮件模板、设置邮箱授权 |
| Agents（Agent 配置） | 配置"邮件分析 Agent"的分析策略、优先级判定逻辑、回复风格 |
| Logs（日志） | 查看邮件 Agent 的执行日志、Token 消耗、异常告警 |

#### 优缺点

| 优点 | 缺点 |
|------|------|
| 最彻底地融入 3.0 架构，无概念冗余 | 实现复杂度最高，涉及三个视图同时开发 |
| 每个角色看到的都是自己需要的 | 邮件处理的"完整体验"被打散，缺少统一管理感 |
| 邮件能力成为平台级能力，可被任何 Agent 调用 | 对于邮件重度用户，inline 操作可能不够 |
| 维护者可以精细控制邮件 Agent 行为 | MVP 阶段如果只做一线员工视图，其他角色的邮件能力要延后 |
| 自动化规则由专业维护者管理，更可控 | 一线员工失去自配置自动化规则的灵活性 |

#### 适用场景

适合**D 系统 3.0 三个角色视图都要在近期落地**的情况，且团队有能力同时推进多个模块。

## 三、方案对比总结

| 维度 | 方案 A：融入 Worker | 方案 B：专项工作台 | 方案 C：拆散到角色 |
|------|-------------------|-----------------|-----------------|
| 邮件入口 | 无独立入口，Worker 视图中的一种任务 | Worker 中的任务卡片点击进入 | 无独立入口，各角色视图中按需呈现 |
| 交互范式 | 完全复用 Worker 范式 | Worker 范式 + 邮件场景增强 | Worker 范式（一线）+ Dashboard（决策）+ Works（维护） |
| 邮件处理深度 | 浅（适合轻量邮件场景） | 深（完整邮件上下文和操作） | 中（一线浅、维护深） |
| 自动化规则归属 | 维护者 works 模块 | 维护者 works + 一线员工可提需求 | 维护者 works 模块 |
| 与 3.0 框架契合度 | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| 开发成本 | 低 | 中 | 高 |
| 适用岗位范围 | CS 等邮件轻度岗位 | 所有岗位（包括邮件重度岗位） | 所有岗位 + 管理层 |
| MVP 可行性 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ |

## 四、待讨论问题

1. **目标用户优先级**：3.0 MVP 阶段主要服务的是 CS 岗位还是所有岗位？CS 岗位的邮件处理需求有多重？
2. **邮件任务的粒度**：一封邮件 = 一个任务？还是 AI 把多封相关邮件归纳为一个任务？
3. **操作闭环的位置**：邮件回复操作是在 Worker 视图内完成（inline），还是允许跳转到专门的处理界面？
4. **自动化规则的配置权**：一线员工是否有权自行配置邮件规则，还是统一由维护者管理？
5. **邮件助手的优先级**：在 3.0 的 roadmap 中，邮件助手排在什么位置？是紧跟 CS 工作台之后，还是后续规划？
